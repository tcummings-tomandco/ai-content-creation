# Routine: write the next Tom & Co article and push to Storyblok

This is the exact prompt fired by the scheduled Claude Routine. It runs end-to-end with no human in the loop until the moment the draft story lands in Storyblok, where Tom McCaul reviews and publishes.

Paste this prompt verbatim into the Claude Routine settings (Tuesday and Thursday 06:30 UK time). It also works in an ad-hoc Claude Code session if you want to run a topic manually.

---

Run the Tom&Co article routine from the repo root.

Read routines/run.md for the full instructions.
Read .claude/skills/tomandco-article/SKILL.md and all supporting files in the same directory: voice-guide.md, checklist.md, stat-sourcing.md, format-mapping.md.

Run: python3 scripts/pick_next_topic.py

Capture the topic's ID, slug-base, and output folder name (`YYYY-MM-DD-{slug}`) from the picker output. The picker prints the full row including ID, Topic, Cluster, Priority, and Quick win.

Execute the full SKILL.md run order (sections 1–7) for that topic. Do not skip the QA gate. Specifically:

1. Load roadmap row + stat bank + voice samples.
2. Research against UK primary sources, target 5–10 citable datapoints.
3. Plan the structure (answer paragraph, H2 questions, comparison table position, internal links).
4. Draft to the cluster-appropriate format. British spellings, no em dashes, no banned words.
5. Generate the JSON-LD block inline.
6. Self-review against the 10-element checklist in checklist.md.
7. Write outputs to `output/{YYYY-MM-DD}-{slug}/`:
   - article.json
   - preview.html
   - sources.md
   - qa_report.md
   - internal_link_suggestions.md
   - stat_bank_update.json (if Layer 2 calc was used)

Run: python3 scripts/voice_check.py output/{folder}/article.json

If any FAIL, iterate the draft and re-run until CLEAN. Voice gate CLEAN is required before proceeding to publish.

Run: python3 scripts/publish_preview.py output/{folder}

This rebuilds the docs/ Pages preview library so the new draft is visible at the engine's GitHub Pages URL while it sits in Storyblok awaiting publish.

If QA passes, push the article to Storyblok as a draft:

```bash
python3 scripts/push_to_storyblok.py output/{folder}/article.json \
  --parent-slug ai-agency-consultancy/blog \
  --category Strategy
```

(`--category Strategy` is a temporary override until the "AI" option is added to the `ai_blog_post` schema in the website repo and seeded. Once that lands, drop the flag and the engine uses the article's own "AI" category.)

Capture the values from the script's STDOUT footer:
- `STORYBLOK_STORY_ID=...`
- `STORYBLOK_FULL_SLUG=...`

Then mark the topic done in the local roadmap:

```bash
python3 scripts/mark_topic_done.py {ID} \
  --slug {slug} \
  --storyblok-id {STORYBLOK_STORY_ID} \
  --storyblok-full-slug {STORYBLOK_FULL_SLUG}
```

Then commit and push the engine-state updates to the engine repo:

```bash
gh auth switch --user tcummings-tomandco
git config user.email "routine@tomandco.co.uk"
git config user.name "Tom & Co AI Routine"
git add output/{folder}/ data/roadmap.json data/stat_bank.json docs/
git commit -m "article: {slug} — auto-generated by routine, draft in Storyblok"
git push
```

If any step fails, stop and surface the failure in the summary. Do not retry destructively. Do not push to the Tom&Co website storyblok repo — that repo is for schema only; articles go to Storyblok via the Management API.

## Schedule

Cloud-scheduled Claude Routine, runs without the laptop or any local Claude session being on:

- **Tuesday 06:30 UK time** — pick the next P0 / P1 / P2 / P3 topic in queue order, draft, push to Storyblok as draft.
- **Thursday 06:30 UK time** — same.

Cadence matches the playbook (Section 8.1): 2 supporting articles per week, ~100 per year, clearing the 75-topic roadmap in roughly 9 months with room for breaking-news pieces.

For breaking news / fast-react pieces (the fortnightly slot the playbook also calls for), trigger the routine manually with an explicit topic-ID override. The picker accepts `--id N` to force a specific topic.

## Environment

The Claude Routine cloud environment must have these variables set (do not commit them):

- `STORYBLOK_OAUTH_TOKEN` — Storyblok Personal Access Token
- `STORYBLOK_SPACE_ID` — numeric space ID
- `STORYBLOK_REGION` — `eu`
- `STORYBLOK_ACCESS_TOKEN` — preview/public token (not strictly needed by the push script but kept consistent with the website repo)
- `GH_TOKEN` or `GITHUB_TOKEN` — GitHub token with `repo` scope for the tcummings-tomandco/ai-content-creation repo, for the routine's commit-and-push step

Locally, the same variables live in `~/Desktop/T&Co/ai-content-creation/variables.env` (gitignored). The push script reads from that file if env vars are not already set.

## Failure modes the routine handles

1. **Voice gate FAIL** — iterate the draft, do not push to Storyblok. If three iterations all fail, stop and surface the specific failure.
2. **Storyblok 4xx** — script raises SystemExit with the error body. Routine fails, no engine-state changes committed.
3. **Duplicate slug** — the push script refuses to overwrite an existing story. Manual intervention required: either delete the previous draft in Storyblok or rename the article slug.
4. **Roadmap empty** — picker exits non-zero. Routine surfaces the message; no more topics to write, raise with Tom.

## Failure modes the routine does NOT handle

- **Stale credentials** — surface the 401, do not retry.
- **GitHub commit conflict** — push fails; routine surfaces the failure. Manual resolution required.
- **Storyblok schema drift** (e.g. someone adds a required field to ai_blog_post without updating the push script) — surface the 422, do not retry.
